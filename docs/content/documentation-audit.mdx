**Why:** Chatons already had memory summarization and an automation runtime, but it did not proactively connect repeated user behavior to automation opportunities.

### Base URL probing now prefers generation-compatible endpoints

**Issue:** Some provider setups passed onboarding tests but failed at runtime with errors like `404 Cannot POST /chat/completions` because base URL probing could accept a candidate that only worked for `/models`.

**Change:** Updated provider base URL probing to score candidates using `/models` plus generation endpoints (`/chat/completions`, `/responses`) and pick the best compatible candidate.

**Why:** Prevents selecting non-functional root URLs (often missing `/v1`) that appear reachable during setup but fail when starting a conversation.

**Files updated:**

- `electron/ipc/workspace-pi.ts` -- Probe scoring logic for base URL candidates
- `docs/content/developer-guide.mdx` -- Documented updated probing behavior
- `docs/content/pi-integration.mdx` -- Added integration-level note on probing behavior
- `AGENTS.md` -- Updated maintainer guidance and troubleshooting notes

### Preserved provider model lists when credentials live in auth.json

**Issue:** Custom providers added from the packaged app could be discovered successfully, then later appear as `0 models` because a normalization pass removed `provider.models` from `models.json` whenever `apiKey` was absent there.

**Root cause:** Chatons migrates API keys into `auth.json`, but the normalizer still treated the absence of `apiKey` inside `models.json` as a signal to delete the provider's `models` array. That made model visibility depend on duplicate credential storage and disproportionately affected packaged-app refresh flows.

**Change:** Stopped deleting `provider.models` during `models.json` normalization when credentials are stored in `auth.json`. Added targeted persistence logs so packaged-app runs can confirm the final stored `baseUrl` and model count per provider.

**Why:** Provider metadata and credentials are separate concerns. Once discovery succeeds, `models.json` must remain authoritative for model visibility regardless of where credentials are stored.

**Files updated:**

- `electron/ipc/workspace-pi.ts` -- Removed destructive normalization of provider model lists
- `electron/ipc/workspace-handlers.ts` -- Added persistence logging for discovered providers
- `docs/content/developer-guide.mdx` -- Clarified `models.json` vs `auth.json` responsibilities
- `docs/content/pi-integration.mdx` -- Documented that model lists persist independently of credential storage
- `AGENTS.md` -- Added maintainer note about preserving provider models during credential migration

### Stabilized packaged-app persistence for custom providers

**Issue:** The installed app still wrote some custom providers with a root `baseUrl` and no `models`, even though the packaged main-process code could successfully resolve `/v1` and discover models when tested directly.

**Root cause:** The custom-provider settings UI persisted provider edits too eagerly, including writes triggered from partially edited in-memory state. That made custom providers more vulnerable than OAuth presets to stale renderer snapshots and out-of-order saves.

**Change:** Updated the provider settings UI to keep text edits local until blur/apply, and to resolve the provider `baseUrl` before persisting a custom provider snapshot. Also captured `console.info` in Electron logs so packaged-app diagnostics include the provider persistence trail.

**Why:** Custom provider persistence now goes through a coherent snapshot instead of a stream of intermediate writes, which reduces the chance of writing a provider entry without resolved `baseUrl` or discovered `models`.

**Files updated:**

- `src/components/sidebar/settings/sections/ProvidersModelsSection.tsx` -- Deferred custom-provider saves to coherent snapshots and normalized `baseUrl` before persistence
- `electron/lib/logging/log-manager.ts` -- Added `console.info` capture to packaged-app logs

### Switched packaged custom-provider probing to Node HTTP transport

**Issue:** Even after fixing persistence and logging, packaged Chatons still failed to discover models for custom local-network providers like `multicodex` and `litellm`. Logs showed the main process trying the correct candidates (`http://host`, `http://host/v1`) but every `fetch` attempt failed, while the same installed binary could reach those endpoints in terminal mode.

**Root cause:** Main-process `fetch` inside the normal packaged Electron runtime was not reliable for those HTTP local-network endpoints. This was runtime-specific: the same endpoint worked in development, in terminal-mode checks, and with manual shell requests.

**Change:** Replaced main-process provider probing, model discovery, and connection tests with a dedicated Node `http/https` client instead of `fetch`.

**Why:** Provider discovery must be deterministic across dev and packaged runs. Using Node's HTTP stack removes the Electron-runtime-specific divergence for custom HTTP providers.

**Files updated:**

- `electron/ipc/workspace-pi.ts` -- Replaced provider probe/discovery transport with Node `http/https`
- `docs/content/developer-guide.mdx` -- Documented the transport choice for custom providers
- `docs/content/pi-integration.mdx` -- Added integration note for packaged local-network providers
- `AGENTS.md` -- Added maintainer note about custom-provider transport behavior

### Declared macOS local-network usage for LAN providers

**Issue:** Packaged macOS builds could still fail against custom providers hosted on LAN IPs (for example `192.168.x.x`) while public HTTPS providers continued to work. Logs showed repeated failures only for local-network HTTP providers.

**Root cause:** The macOS app bundle did not declare `NSLocalNetworkUsageDescription`, which can prevent or destabilize access to local-network services from the packaged app even when ATS exceptions are present.

**Change:** Added `NSLocalNetworkUsageDescription` to the packaged macOS app metadata.

**Why:** Self-hosted AI gateways and OpenAI-compatible LAN endpoints are a core custom-provider use case. The app must explicitly declare local-network access to behave consistently on modern macOS.

**Files updated:**

- `package.json` -- Added macOS `extendInfo.NSLocalNetworkUsageDescription`

### Fixed critical requirement sheet documentation and iframe communication issue

**Issue:** The requirement sheet documentation contained incorrect code examples that would fail silently. Extension developers were instructed to call `window.chaton.requirementSheet.confirm()` and similar functions, but these don't exist in the sandboxed iframe context. This caused tokens to be saved but the confirmation to fail, showing "Failed to save token. Please try again."

**Root Cause:** 
- Requirement sheets render as sandboxed iframes (`allow-scripts allow-same-origin allow-forms`)
- Sandboxed iframes cannot access parent window APIs directly
- The only valid communication mechanism is `window.parent.postMessage()`
- Documentation contradicted this in the "Available methods" section
- All 6 built-in extensions (Vercel, GitHub, Linear, Zapier, Coolify, n8n) had this bug

**Solution:**

1. **Fixed all extension handlers** to use postMessage instead of non-existent function calls:
   - `/Users/thibaut/.chaton/extensions/@thibautrey/chatons-extension-vercel/handler.js`
   - `/Users/thibaut/.chaton/extensions/@thibautrey/chatons-extension-github/handler.js`
   - `/Users/thibaut/.chaton/extensions/@thibautrey/chatons-extension-linear/handler.js`
   - `/Users/thibaut/.chaton/extensions/@thibautrey/chatons-extension-zapier/handler.js`
   - `/Users/thibaut/.chaton/extensions/@thibautrey/chatons-extension-coolify/handler.js`
   - `/Users/thibaut/.chaton/extensions/@thibautrey/chatons-extension-n8n/handler.js`

2. **Rewrote requirement sheet documentation** (`docs/content/extensions/requirement-sheets.mdx`) to:
   - Remove misleading function call examples
   - Add a **critical warning** section emphasizing that iframes are sandboxed
   - Show correct `postMessage` usage prominently with ✅/❌ examples
   - Added a new "Common Mistakes" section with practical examples
   - Updated "How It Works" to mention postMessage explicitly
   - Fixed the "Complete Example" to use postMessage in JavaScript functions
   - Updated constraints section to explicitly mention the sandboxing limitation

**Why:** Extension developers need crystal-clear documentation about iframe sandboxing constraints. Silent failures (token saves that work but appear to fail due to callback errors) are particularly confusing. The documentation now frontloads this limitation.

**Files modified:**

## March 11, 2026 (Session 4)

### Safe Git feature hardening and UX guardrails

**Change:** Improved the reliability of Chatons Git/worktree behavior without expanding into risky Git-client features.

**Why:** The Git feature already exposed status, worktrees, commit, pull/push, and merge actions, but a few implementation details were unsafe or misleading: worktree repositories were not always detected correctly, commit-message generation ignored the actual diff, diff stats were inflated, base branch handling was hardcoded to `main`, and automatic merge could fall back to a synthetic file-copy flow.

**Solution:**

- Updated Git repository detection to support both `.git` directories and `.git` files, which is required for linked worktrees
- Added default-branch resolution in the Git service instead of hardcoding `main`
- Prefer native Git for current-branch lookup, checkout, pull, and push when available
- Changed AI commit-message generation to include real staged/unstaged diffs and status output
- Fixed diff statistics parsing to count actual added/removed lines rather than hunk ranges
- Replaced synthetic worktree merge behavior with a native-Git-only merge path; when native Git is unavailable, the UI now surfaces that limitation instead of attempting a fake merge
- Exposed native Git availability in worktree info so the renderer can disable unsupported actions and explain why

**Files modified:**

- `electron/lib/git/git-service.ts` -- Worktree-safe repo detection, default branch resolution, more native Git delegation
- `electron/ipc/workspace.ts` -- Safer merge logic, real commit-message inputs, corrected diff stats, dynamic base branch handling
- `src/components/shell/Topbar.tsx` -- Native Git status surfaced in source control UI; merge button guarded when unsupported
- `src/types/global.d.ts` -- Extended worktree Git info typing with native Git capability flag
- `docs/content/user-guide.mdx` -- Updated worktree behavior and limitations
- `docs/content/developer-guide.mdx` -- Documented native-Git requirement for automatic merge

**Safety posture:** Deliberately did not implement risky recommendations such as synthetic conflict resolution, pseudo-merge fallbacks, or broad new destructive actions in the main repository UI.

## March 11, 2026 (Session 3)

### Enhanced manifest file detection for extensions

**Change:** Modified `electron/extensions/runtime/manifest.ts` to support both `manifest.json` and `chaton.extension.json` manifest filenames.

**Why:** Third-party extensions published to npm were using `manifest.json` as the standard filename (aligning with common extension naming conventions), while Chatons was only looking for `chaton.extension.json`. This caused npm-published extensions with `manifest.json` to fail to load with "Package not found in node_modules" errors, even though the package was installed correctly.

**Solution:**

- Updated `readManifestFromExtensionDir()` in `manifest.ts` to iterate through a priority list of manifest filenames: `['manifest.json', 'chaton.extension.json']`
- `manifest.json` is preferred (tried first); `chaton.extension.json` is the legacy fallback
- Both formats are fully supported and validated
- Existing builtin and user extensions using `chaton.extension.json` continue to work without changes

**Files modified:**

- `electron/extensions/runtime/manifest.ts` -- Added manifest filename iteration logic
- `docs/content/extensions/index.mdx` -- Updated documentation to mention both filename formats

**Build verification:** Compilation successful, no TypeScript errors.

**Compatibility:** Fully backward compatible. All existing extensions continue to work.

**Recommendation for extension developers:** Use `manifest.json` when publishing new extensions to npm for better alignment with standard extension conventions.
