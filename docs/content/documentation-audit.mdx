# Documentation Audit

## 2026-03-20 (Latest)

### Added cloud subscriptions, per-user cloud quotas, and an admin account panel

**Change:** Added cloud-account subscription state, required an active subscription for cloud bootstrap/runtime access, enforced concurrent remote-session quotas server-side, and exposed a new Cloud settings panel in the desktop app with account details, usage, editable plan parameters, and an admin-only user management view. The first created cloud user is now marked admin by default.

**Why:** Cloud access and remote-runtime capacity need to be owned by the cloud account instead of the local desktop. Enforcing quotas only in the renderer would be bypassable and would not reflect the real server-side resource policy. The desktop also needs a built-in place for users to inspect their subscription and for admins to manage both users and plan parameters before billing is implemented.

**Verified files:**
- `packages/domain/index.ts` -- Added shared cloud user, subscription, and usage contracts
- `packages/protocol/index.ts` -- Added account/admin protocol payloads and runtime-session create responses with quota errors
- `apps/cloud-api/server.ts` -- Added in-memory cloud users, default-admin assignment, account/admin endpoints, and subscription-gated bootstrap
- `apps/runtime-headless/server.ts` -- Enforces per-user parallel cloud session limits from dynamic plan parameters sent by the cloud control plane
- `electron/ipc/workspace-handlers.ts` -- Fetches cloud account/admin data, forwards subscription tier to runtime creation, and exposes account/admin IPC handlers
- `electron/preload.ts` -- Exposed cloud account/admin methods to the renderer
- `src/services/ipc/workspace.ts` -- Added renderer IPC wrappers for cloud account/admin actions
- `src/features/workspace/types.ts` -- Added cloud account, subscription, and usage renderer types
- `src/features/workspace/store/provider.tsx` -- Hydrates cloud account/admin state and removed the old renderer-side cloud-runtime placeholder block
- `src/components/sidebar/settings/sections/CloudSection.tsx` -- Added desktop account/subscription/admin UI

- `docs/content/pi-integration.mdx` -- Documented subscription-gated cloud runtime behavior
- `docs/content/developer-guide.mdx` -- Documented cloud-account state in desktop/runtime architecture

### Added authenticated cloud bootstrap sync for desktop sessions

**Change:** Connected the browser-based desktop cloud login to an authenticated cloud bootstrap fetch, added local cloud-project upsert logic, and made startup hydration resync connected cloud instances from the stored desktop session. The local projects schema was also migrated so cloud projects can persist without any local `repo_path`.

**Why:** Desktop cloud login was already able to store a session locally, but the session was not yet used to materialize cloud projects in the workspace. That left the UI connected but functionally empty. Cloud projects also need to exist without any local repository path because they do not run on local Pi.

**Verified files:**

- `packages/sdk-cloud/index.ts` -- Added bearer-token support for cloud API calls
- `docs/content/pi-integration.mdx` -- Clarified that cloud projects do not use local Pi and hydrate from cloud bootstrap
- `docs/content/developer-guide.mdx` -- Clarified local vs cloud runtime ownership during workspace hydration

### Added authenticated realtime websocket flow for connected cloud instances

**Change:** Added a token-gated websocket endpoint to `cloud-realtime`, connected desktop main-process listeners to that endpoint after cloud auth/bootstrap, and forwarded realtime cloud status changes to the renderer for workspace rehydration.

**Why:** Cloud projects need a live connection-health channel that is independent from the local Pi runtime and that fits the Kubernetes deployment model where API and realtime workloads scale separately.

**Verified files:**
- `apps/cloud-realtime/server.ts` -- Added token validation and websocket status stream
- `electron/ipc/workspace-handlers.ts` -- Added main-process realtime socket lifecycle and status propagation
- `electron/preload.ts` -- Exposed realtime cloud event subscription to the renderer
- `src/services/ipc/workspace.ts` -- Added renderer-side realtime listener bridge
- `src/features/workspace/store/provider.tsx` -- Rehydrates workspace state on cloud realtime events
- `docs/content/cloud-kubernetes.mdx` -- Documented the split HTTP/bootstrap and websocket workloads

### Added initial remote runtime proxy for cloud conversations

**Change:** Added conversation-level cloud runtime persistence, a minimal `runtime-headless` session API, and main-process routing so `pi:startSession`, `pi:sendCommand`, `pi:getSnapshot`, and `pi:stopSession` use the remote runtime for cloud conversations instead of the local Pi runtime.

**Why:** Cloud projects were visible and authenticated, but cloud conversations still could not execute. The desktop now treats cloud conversations as remote sessions while preserving the existing local Pi path for local conversations.

**Verified files:**
- `electron/db/migrations/026_cloud_conversation_runtime.sql` -- Added conversation runtime-location and remote-session persistence
- `electron/db/repos/conversations.ts` -- Persists cloud runtime session ids on conversations
- `apps/runtime-headless/server.ts` -- Added minimal create/command/snapshot/delete runtime session endpoints
- `electron/ipc/workspace-handlers.ts` -- Routes `pi:*` handlers to the remote runtime for cloud conversations
- `electron/ipc/workspace.ts` -- Exposes runtime location in the workspace payload
- `src/features/workspace/types.ts` -- Adds conversation runtime-location typing

### Added realtime conversation event flow between runtime-headless and cloud-realtime

**Change:** Added event publishing from `runtime-headless` to `cloud-realtime`, extended the realtime protocol with conversation events, and forwarded those events back into the desktop `pi:event` pipeline so cloud conversations can update incrementally without a full workspace snapshot after every command.

**Why:** The earlier cloud runtime proxy still depended on snapshot refreshes after commands. A conversation-grade event path is required to approach the local Pi runtime behavior and to make the cloud architecture genuinely realtime.

**Verified files:**
- `packages/protocol/index.ts` -- Added realtime conversation event payloads
- `apps/cloud-realtime/server.ts` -- Added event ingestion and fan-out to websocket subscribers
- `apps/runtime-headless/server.ts` -- Publishes runtime-status and message-update events
- `electron/ipc/workspace-handlers.ts` -- Maps realtime conversation events back into `pi:event`
- `src/features/workspace/store/provider.tsx` -- Routes cloud conversation events through `applyPiEvent`

### Added richer cloud conversation lifecycle events

**Change:** The cloud runtime now emits a more complete event lifecycle for prompts, including `tool_execution_start`, `tool_execution_end`, `message_update`, `agent_end`, and final `runtime_status` events, and it now publishes them under the actual cloud instance id associated with the desktop session.

**Why:** A cloud conversation needs to look and behave like a real runtime session inside the existing renderer pipeline. Without consistent instance routing and richer lifecycle events, the desktop could not reuse the current Pi event handling correctly.

**Verified files:**
- `apps/runtime-headless/server.ts` -- Publishes richer lifecycle events and uses the session cloud instance id
- `electron/ipc/workspace-handlers.ts` -- Passes the real cloud instance id when creating runtime sessions

### Moved cloud project and conversation creation behind cloud-api

**Change:** Added minimal per-session cloud state to `cloud-api`, including server-owned project and conversation creation endpoints, and updated the desktop main process to create cloud projects and cloud conversations through the API before resyncing local state from bootstrap.

**Why:** Projects and conversations that belong to the cloud need a server-side source of truth. Creating them only in the local desktop cache would make multi-user collaboration and cross-device sync impossible.

**Verified files:**
- `apps/cloud-api/server.ts` -- Added in-memory project and conversation state plus create endpoints
- `packages/protocol/index.ts` -- Added cloud project and conversation create request/response types
- `electron/db/repos/conversations.ts` -- Added conversation upsert helper for bootstrap sync
- `electron/ipc/workspace-handlers.ts` -- Creates cloud entities via `cloud-api` and resyncs from bootstrap

### Added server-owned message history for cloud conversations

**Change:** Added per-conversation message storage to `cloud-api`, made `runtime-headless` persist message history back to the API, and taught the desktop message-cache path to fetch cloud conversation history from the server before serving the local cache.

**Why:** Cloud conversations need to survive desktop restarts and runtime session recreation. Without server-owned message history, a cloud conversation could authenticate correctly but still reopen with an empty transcript.

**Verified files:**
- `packages/protocol/index.ts` -- Added cloud conversation message types
- `apps/cloud-api/server.ts` -- Added conversation message read/write endpoints
- `apps/runtime-headless/server.ts` -- Persists message history to `cloud-api`
- `electron/ipc/workspace-handlers.ts` -- Fetches cloud conversation history into the local message cache

### Changed conversation memory injection to separate steer context

**Change:** Reworked conversation-start memory injection so relevant memories are sent as a separate runtime `steer` message before the first prompt instead of being appended to the user's initial message. The injected memory text was also reframed as low-confidence internal background context that must yield to the active request, repo state, and tool evidence.

**Why:** Appending memory summaries to the first user message polluted user intent and could mislead the agent into treating stale memories as current instructions. Separating the memory into steer context preserves message boundaries and makes the memory semantically weaker and easier for the model to ignore when contradicted.

**Verified files:**
- `electron/ipc/workspace-handlers.ts` -- Switched first-turn memory injection from prompt concatenation to a separate `steer`
- `electron/extensions/runtime/memory-lifecycle.ts` -- Reworded memory context to be explicitly low-confidence and non-instructional
- `AGENTS.md` -- Documented the maintainer rule for memory injection semantics
- `docs/content/pi-integration.mdx` -- Documented runtime memory injection behavior at conversation start
- `docs/content/developer-guide.mdx` -- Documented why memory is injected as steer instead of user text

### Added desktop cloud browser-auth callback flow

**Change:** Wired the desktop cloud connection flow to launch the system browser, receive a `chatons://cloud/auth/callback` deep link, complete the cloud session locally, and refresh the desktop cloud state. The cloud docs were updated to reflect this browser-based desktop auth model.

**Why:** The intended UX is not a manual URL prompt or in-app embedded login. Desktop Chatons should open `cloud.chatons.ai` in the user browser, detect the callback when login finishes, and keep the session locally for future cloud use.

**Verified files:**
- `electron/main.ts` -- Added deep-link handling for `chatons://cloud/auth/callback`
- `electron/preload.ts` -- Exposed cloud auth start/complete APIs and cloud auth callback listener
- `electron/ipc/workspace-handlers.ts` -- Added `cloud:startAuth` and `cloud:completeAuth`
- `electron/db/migrations/024_cloud_instance_sessions.sql` -- Added persisted cloud session fields
- `electron/db/repos/cloud-instances.ts` -- Added OAuth state and desktop session persistence helpers
- `src/features/workspace/store/provider.tsx` -- Switched cloud connection to browser-based auth completion flow
- `docs/content/cloud-architecture.mdx` -- Clarified the browser-to-desktop callback flow

## 2026-03-20

### Hardened conversation worktrees, Git status caching, and window/runtime cleanup

**Change:** Tightened project-thread worktree lifecycle, corrected Git status cache semantics, switched worktree removal to prefer native `git worktree remove --force`, cleared stale main-window references on real window destruction, and disposed workspace runtime event subscriptions during shutdown. Pi auth/config writes were also moved to atomic JSON writes for consistency with the rest of the managed Pi directory.

**Why:** The previous worktree flow could persist an empty fallback directory as a valid conversation worktree after creation failures, direct directory deletion could leave orphaned native Git worktree metadata behind, and the Git status cache could return incorrect staged/unstaged answers depending on call order. Separately, the main process could retain a destroyed window reference while the app stayed alive in the background, and workspace handler shutdown did not release its Pi runtime subscription.

**Verified files:**
- `electron/ipc/workspace.ts` -- Worktree creation now fails closed, cleanup matches the real worktree directory layout, and async Git repo checks are awaited correctly
- `electron/ipc/workspace-handlers.ts` -- Worktree removal now receives the parent repo path and runtime event subscriptions are disposed on shutdown
- `electron/lib/git/git-service.ts` -- Reworked status caching, added native worktree removal helper, and exported parsing helpers for tests
- `electron/lib/git/git-service.test.ts` -- Added tests covering independent staged vs unstaged status parsing
- `electron/db/repos/conversations.ts` -- Allows explicit nullable runtime-field clears instead of only COALESCE-style updates
- `electron/ipc/workspace-pi.ts` -- Uses atomic JSON writes for `auth.json` sync/bootstrap paths
- `electron/main.ts` -- Registers window IPC once and clears `mainWindow` on actual window destruction
- `docs/content/pi-integration.mdx` -- Documented strict worktree persistence and native worktree removal behavior
- `docs/content/developer-guide.mdx` -- Documented stricter worktree/runtime cleanup and destroyed-window handling

## 2026-03-17 (Latest)

### Fixed CronJob removeAllListeners() TypeError in cron scheduler

**Change:** Removed calls to `CronJob.removeAllListeners()` in `electron/extensions/runtime/cron-scheduler.ts` `stop()` and `stopAll()` methods.

**Why:** The `cron` package v3.5.0's `CronJob` class does not have a `removeAllListeners()` method. This method existed in earlier versions but was removed. Calling it caused a `TypeError: task.job.removeAllListeners is not a function` during extension runtime shutdown, preventing graceful cleanup of cron scheduler automation tasks.

**Impact:** Cron-scheduled automation tasks now properly stop and cleanup without errors during app shutdown. The `stop()` method still calls `task.job.stop()` to properly halt the job; EventEmitter listener cleanup is no longer needed or attempted.

**Verified files:**
- `electron/extensions/runtime/cron-scheduler.ts` -- Removed two `task.job.removeAllListeners()` calls from `stop()` and `stopAll()` methods
- `dist-electron/extensions/runtime/cron-scheduler.js` -- Compiled output verified to have no `removeAllListeners` references

## 2026-03-17

### Routed GitHub Copilot to Anthropic message shaping

**Change:** Switched the built-in GitHub Copilot provider preset and OAuth provider bootstrap from Pi's `openai-completions` transport to `anthropic-messages`.

**Why:** Runtime verification showed that Copilot Claude models succeed with the same token, base URL, and required Copilot headers when they receive Anthropic-style payloads, while the OpenAI-compatible chat-completions path still returned `400 Bad Request` for minimal prompts in Chatons conversations. The durable fix is therefore to keep the Copilot endpoint and auth flow, but use the request format Copilot reliably accepts for these Claude-backed models.

**Verified files:**
- `electron/ipc/workspace-handlers.ts` -- Bootstraps OAuth-backed GitHub Copilot providers with `anthropic-messages`
- `src/features/workspace/provider-presets.ts` -- Defaults the GitHub Copilot preset to `anthropic-messages`
- `src/components/model/ProviderSetupForm.tsx` -- Allows `anthropic-messages` in provider setup forms
- `src/components/sidebar/settings/sections/ProvidersModelsSection.tsx` -- Allows `anthropic-messages` in provider settings state
- `src/components/onboarding/OnboardingFlow.tsx` -- Allows `anthropic-messages` in onboarding provider setup state
- `docs/content/pi-integration.mdx` -- Documented the GitHub Copilot transport change

**Change:** Reworked conversation auto-title refinement so it uses a hidden ephemeral Pi runtime session, matching the memory summarization pattern, instead of invoking the Pi CLI directly for a one-off prompt.

## 2026-03-14

### Added extension registry npm auto-discovery

**Change:** Updated the standalone `extension-registry` sync flow so it can auto-discover published npm extensions instead of relying only on manually curated package names in `data/registry.json`.

**Why:** Newly published Chatons npm extensions were valid and publicly available on npm, but they never appeared in `marketplace.chatons.ai` unless someone manually added the package name to the registry source file. That created drift between npm and the marketplace catalog.

**Verified files:**
- `extension-registry/lib/discovery.ts` -- Added npm search-based discovery and candidate verification
- `extension-registry/lib/sync.ts` -- Merged curated and auto-discovered npm packages during catalog generation
- `extension-registry/lib/sync-runner.ts` -- Enabled discovery for local sync runs
- `extension-registry/lib/dev-server.ts` -- Enabled discovery for local manual syncs
- `extension-registry/api/cron/sync.ts` -- Enabled discovery for production cron syncs
- `extension-registry/lib/types.ts` -- Added `autoDiscoveredNpm` support to the registry source shape
- `extension-registry/README.md` -- Documented the new discovery behavior and curated fallback role

**Notes:**
- Builtin extensions remain explicitly declared in `data/registry.json`.
- Curated npm entries still serve as a fallback for packages that npm search has not indexed yet.
- Auto-discovered packages are merged at sync time and are not written back into `data/registry.json`.
