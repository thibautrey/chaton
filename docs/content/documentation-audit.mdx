# Documentation Audit

## 2026-03-20 (Latest)

### Tuned the Kubernetes base manifests for a true single-node profile

**Change:** Reduced the default replica count for `cloud-api`, `cloud-realtime`, and `runtime-headless` from `2` to `1`, lowered HPA minimum replicas to `1`, and reduced stateless-service CPU and memory requests/limits so the base package wastes less capacity on a single-node cluster while keeping PostgreSQL and Redis persistence intact.

**Why:** The earlier base manifests technically ran on one node, but they still reserved capacity for a multi-replica baseline that cannot provide real high availability on a single-node cluster. Lowering the baseline replica count and resource requests makes the default deployment shape match the actual topology the cluster is running.

### Added admin-managed complimentary cloud subscriptions with optional expiry

**Change:** Extended the cloud account model so admins can now allocate a complimentary subscription plan to any user either indefinitely or for a given number of days. The cloud control plane now persists the optional grant window, treats the active grant as the effective subscription for account payloads and quota enforcement, exposes a dedicated admin endpoint for grant assignment, and the desktop Cloud settings panel now includes UI to assign those grants per user.

**Why:** The previous admin tooling only let operators edit a user's baseline plan directly, which did not model “free access without payment for a limited time” as a first-class concept. Complimentary grants need their own duration-aware control-plane state so internal/admin allocations do not require billing changes and can expire automatically without manual cleanup.

### Hardened Kubernetes deployment kubeconfig validation in GitHub Actions

**Change:** Updated the cloud deployment workflow so it now verifies that the `KUBE_CONFIG` secret is present, decodes it with `printf` instead of `echo`, prints the configured kube contexts, and runs an explicit cluster-connectivity check before attempting namespace creation. The Kubernetes deployment docs now also call out `Unable to connect to the server: EOF` as a kubeconfig or API reachability failure mode rather than a namespace issue.

**Why:** The previous workflow went straight from kubeconfig decoding to `kubectl get namespace`, which made connectivity failures look like a namespace bootstrap problem. In practice, an `EOF` from the API server at that point means the runner could not establish a usable Kubernetes session, so the job should fail earlier and more clearly.

**Verified files:**
- `.github/workflows/deploy-cloud.yml` -- Added kubeconfig presence checks, context output, and preflight cluster connectivity validation
- `docs/content/cloud-kubernetes.mdx` -- Documented the `EOF` failure mode and likely kubeconfig causes

### Made Chatons Cloud its own OIDC provider for desktop login

**Change:** Replaced the ad-hoc desktop cloud auth flow with a built-in OIDC authorization-code + PKCE implementation in `cloud-api`. The service now exposes discovery, JWKS, authorization, token, and userinfo endpoints, persists authorization requests with PKCE metadata and user binding, signs ID tokens, and the Electron desktop now performs a standards-based browser redirect plus token exchange against those endpoints.

**Why:** The previous desktop auth flow looked OIDC-shaped from the outside but was still a custom `/desktop/auth` plus exchange endpoint, which meant Chatons could not actually act as its own issuer. Making the cloud control plane a real OIDC provider removes that mismatch and lets the desktop connect through a standard authorization-code flow rooted at the Chatons Cloud base URL.

**Verified files:**

### Switched bundled cloud data services to persistent single-node stateful workloads

**Change:** Reworked the scaffolded Kubernetes PostgreSQL and Redis manifests from stateless `Deployment`s into single-replica `StatefulSet`s with PVC-backed storage, moved PostgreSQL bootstrap password injection to `chatons-cloud-secrets`, and updated the Kubernetes deployment docs plus example secret to reflect the new password key and the built-in Chatons OIDC issuer settings.

**Why:** The previous base manifests lost PostgreSQL and Redis data whenever a pod was recreated, which is the wrong default even for a one-node bootstrap cluster. The updated manifests keep state across restarts and the docs now align the secret shape with the self-hosted OIDC-capable cloud stack.

### Hardened cloud auth, quota enforcement, runtime ownership, and Kubernetes safeguards

**Change:** Reworked the cloud services so plan definitions and desktop-auth request state are now durable in `cloud-api`, added internal control-plane access endpoints for runtime/realtime authorization, moved cloud runtime quota enforcement to server-derived grants instead of client-supplied headers, required bearer ownership on runtime session reads/commands/deletes, protected realtime event publishing behind the shared internal service token, added bounded request-body handling and websocket/token cleanup, and extended the Kubernetes base with TLS, ingress rate limits, topology spread, pod disruption budgets, and a baseline network policy.

**Why:** The earlier cloud implementation still trusted the desktop client for quota values, allowed unauthenticated realtime event injection, and exposed runtime-session APIs without validating that the caller owned the session. That was not safe for multi-tenant cloud usage and also left the manifests behind the runtime assumptions. The hardening changes move authorization back to the control plane and make the deployment model match the code.

### Made cloud realtime multi-replica aware for Kubernetes

**Change:** Reworked `cloud-realtime` so websocket tokens are now bound explicitly to the desktop cloud-instance id, and added optional Redis-backed Pub/Sub fan-out for realtime events. The desktop main process now includes the local cloud instance id when minting a realtime token, and the Kubernetes base manifests now include a Redis deployment plus realtime channel configuration.

**Why:** The previous realtime implementation derived the target cloud instance from the bearer token suffix and only fanned out events from in-process memory. That breaks down once `cloud-realtime` runs with multiple replicas behind Kubernetes because the websocket connection and the event-ingesting pod are no longer guaranteed to be the same process.

### Reused active runtime sessions per cloud conversation

**Change:** Updated `runtime-headless` to look up an already-active session for the same `(conversationId, accessToken)` before creating a new one. If such a session exists, the service now returns it instead of allocating a duplicate runtime.

**Why:** Once runtime session state became shared across pods, the next failure mode was duplicate runtime creation for the same conversation when the desktop retried quickly or multiple pods handled overlapping requests. Reusing the existing active session is a pragmatic first ownership rule and prevents accidental double-counting against parallel-session quotas.

**Verified files:**
- `apps/runtime-headless/store.ts` -- Added lookup of active runtime sessions by conversation and access token
- `apps/runtime-headless/server.ts` -- Reuses an existing active runtime session before creating a new one
- `docs/content/cloud-kubernetes.mdx` -- Documented active-session reuse for cloud conversations

### Added runtime session lease ownership for cloud conversations

**Change:** Extended runtime sessions with an owner id and lease expiration, configured a lease TTL for `runtime-headless`, and renewed that lease during cloud runtime activity. Active-session lookups now ignore expired leases, allowing a new pod to reclaim a conversation after the old owner disappears.

**Why:** Reusing an existing session is not enough if the owning pod crashes without cleanup. A server-side lease is the minimum mechanism needed so conversation ownership can eventually recover instead of getting stuck behind stale active-session records.

**Verified files:**
- `apps/runtime-headless/store.ts` -- Added owner and lease fields plus lease-aware active-session queries
- `apps/runtime-headless/server.ts` -- Assigns, refreshes, and expires conversation session leases
- `k8s/base/configmap.yaml` -- Added runtime lease TTL configuration
- `k8s/base/runtime-headless.yaml` -- Injects runtime lease TTL into the deployment
- `docs/content/cloud-kubernetes.mdx` -- Documented runtime session lease ownership

### Added lease heartbeats and desktop recovery for expired cloud runtime sessions

**Change:** `runtime-headless` now refreshes leases periodically for sessions owned by the current pod, and the desktop main process now clears and reacquires a cloud runtime session if the stored remote session id returns `404` during snapshot or command execution.

**Why:** A lease without heartbeats still expires during healthy long-lived sessions, and desktop state without recovery still gets stuck on stale `cloud_runtime_session_id` values after a pod restart or reclaimed lease. The combination of periodic lease renewal plus client-side reacquisition closes that gap.

**Verified files:**
- `apps/runtime-headless/store.ts` -- Added listing of sessions by owner for lease refresh cycles
- `apps/runtime-headless/server.ts` -- Added periodic lease renewal for owned sessions
- `electron/ipc/workspace-handlers.ts` -- Clears stale cloud runtime session ids and reacquires remote sessions after `404`
- `docs/content/cloud-kubernetes.mdx` -- Documented desktop recovery from stale cloud runtime session ids

### Added realtime replay buffering for cloud reconnects

**Change:** Added a bounded replay buffer in `cloud-realtime`, exposed it through `/v1/realtime/replay`, and made the desktop main process fetch and re-emit those buffered conversation events immediately after the websocket connects.

**Why:** Without replay, a brief websocket outage meant the desktop only saw new events emitted after reconnection. Recent runtime events that happened during the disconnect were only recoverable indirectly through transcript or snapshot paths. A short replay buffer closes that gap for normal reconnect scenarios.

**Verified files:**
- `packages/protocol/index.ts` -- Added the realtime replay response type
- `apps/cloud-realtime/server.ts` -- Added bounded per-instance replay buffering and replay endpoint
- `electron/ipc/workspace-handlers.ts` -- Fetches and reapplies replayed conversation events after realtime connect
- `docs/content/cloud-kubernetes.mdx` -- Documented replay buffering for cloud reconnects

### Moved realtime replay buffering onto Redis when available

**Change:** Extended `cloud-realtime` so its bounded replay buffer is now also written to and read from Redis lists when `REDIS_URL` is configured, while keeping the in-memory fallback for local development. The Kubernetes config now exposes the replay buffer size explicitly.

**Why:** The first replay implementation still lost its buffer when a `cloud-realtime` pod restarted. That is exactly the failure mode where reconnect replay matters most in Kubernetes. Storing the replay buffer in Redis lets the reconnect path survive those pod restarts.

**Verified files:**
- `apps/cloud-realtime/server.ts` -- Stores and reads bounded replay buffers from Redis with in-memory fallback
- `k8s/base/configmap.yaml` -- Added replay buffer size configuration
- `k8s/base/cloud-realtime.yaml` -- Injects replay buffer configuration into the realtime deployment
- `docs/content/cloud-kubernetes.mdx` -- Clarified Redis-backed replay persistence

### Added incremental realtime replay by sequence number

**Change:** Added per-instance monotonically increasing `seq` values to realtime events, extended replay responses with `lastSeq`, and changed the desktop reconnect flow to request replay only after the last seen sequence for that cloud instance.

**Why:** Replaying the last `N` events on every reconnect is simple but noisy and can still reapply already-consumed events repeatedly. Sequence-based replay gives the desktop a stable offset and moves the realtime contract toward a proper resumable event stream.

**Verified files:**
- `packages/protocol/index.ts` -- Extended replay responses with `lastSeq`
- `apps/cloud-realtime/server.ts` -- Assigns per-instance event sequences and filters replay by `afterSeq`
- `electron/ipc/workspace-handlers.ts` -- Tracks last seen realtime sequence per cloud instance and requests incremental replay
- `docs/content/cloud-kubernetes.mdx` -- Documented sequence-based replay behavior

### Realigned cloud control-plane ownership after cloud-api rewrite
