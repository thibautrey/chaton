**Change:** Added an opt-in hybrid harness mode that records which harness candidate started each local conversation, shows the active harness in the conversation UI, lets users vote thumbs-up / thumbs-down after the agent responds, stores those votes in SQLite, exposes aggregate feedback in the Meta-Harness panel, and feeds a bounded human-feedback bonus back into optimizer scoring.

**Why:** The initial Meta-Harness benchmark is too narrow to reflect long-run user satisfaction on real coding work. A hybrid score lets Chatons keep the reproducibility of benchmark evaluation while progressively incorporating direct user judgments about whether the conversation outcome was actually useful.

## 2026-03-30

### Fixed conversation creation / runtime model desynchronization in the composer

**Change:** New conversation creation now always falls back to the saved composer model when no explicit `modelProvider` / `modelId` is supplied by the caller. The composer model synchronization was also tightened so an existing conversation's persisted/runtime model takes precedence over a stale local picker selection.

**Why:** In some flows, the composer visibly showed the previously selected model while the newly created conversation had no persisted model pair, causing Pi to start on the managed default model instead. Because the picker could still prefer its local selection after the runtime started, the UI and the actual model used by Pi could diverge.

**Files updated:**
- `src/features/workspace/store/provider.tsx` -- Added saved-composer-model fallback for global and project conversation creation
- `src/components/shell/composer/useComposerModelState.ts` -- Made persisted/runtime conversation model authoritative over local picker state for existing conversations

## 2026-03-29

### Fixed thinking block stripping for AI title refinement

**Change:** `supprimerBlocsThinking()` in `workspace-title.ts` now matches the full pattern set from `messageParsing.ts:stripThinkingBlocks()`, including the non-whitespace-prefixed Anthropic-style `` pattern. Added whitespace normalization to clean up after block removal.

**Why:** When a model returned reasoning blocks (e.g., ``) before the actual title, the title parser failed because it only handled the whitespace-prefixed variant. This caused AI title refinement to silently fall back to deterministic titles even when a valid short title was present after the thinking block.

**Files updated:**
- `electron/ipc/workspace-title.ts` -- Added missing anthropic-style pattern and whitespace cleanup to `supprimerBlocsThinking()`

## 2026-03-23

### Bundled npm for packaged extension install, update, and publish flows

**Change:** Packaged desktop builds now ship a real `npm` CLI alongside the app resources and run it through Electron's embedded Node runtime (`process.execPath` with `ELECTRON_RUN_AS_NODE=1`) for extension install, update, publish, and sandboxed npm execution. The extension command resolvers were also tightened so they no longer treat the bare string `npm` as if it were a JavaScript entrypoint.

**Why:** The previous packaged-app behavior still depended on the GUI process inheriting a developer shell `PATH`, which is not reliable on macOS and other desktop launches. That produced false `npm command not found` failures even though Electron already embeds Node. Bundling `npm` removes that ambient dependency and keeps extension package-manager behavior consistent between development and compiled apps.

### Reworked memory into a structured cross-runtime system

**Change:** Replaced the old desktop-only, summary-heavy memory path with a shared structured memory model used by both desktop and cloud runtimes. Desktop memory now stores structured fields such as `topicKey`, `confidence`, `reinforcedAt`, `lastUsedAt`, `timesUsed`, provenance, status, and visibility. Retrieval now uses local FTS plus metadata-aware reranking and supports correct mixed-scope search. Cloud memory is now persisted by `cloud-api`, and `runtime-headless` exposes the same memory tools by forwarding requests to authenticated control-plane routes.

**Why:** The previous implementation wrote one summary blob per run, tried to clean duplication later, and diverged between desktop and cloud behavior. The new design stores higher-signal typed memories, keeps retrieval tool-driven, and moves durable cloud memory state into the control plane so memory semantics stay consistent across runtimes.

### Prevented hosted cloud signup from hanging on SMTP timeouts

### Aligned the hosted cloud ingress and public URL guidance with `chatons.ai`

**Why:** Some reasoning-capable models emitted internal thinking text instead of a plain title, causing title sanitization to reject otherwise recoverable outputs and making the fallback warning sound like the refinement path itself was unavailable.

**Verified files:**
- `electron/ipc/workspace-title.ts` -- Strips reasoning blocks before title validation and logs both raw and cleaned previews for unusable outputs
- `electron/ipc/workspace-handlers.ts` -- Clarified the fallback warning when AI title refinement returns no usable title
- `docs/content/pi-integration.mdx` -- Documented reasoning-block stripping for hidden title-generation sessions

## 2026-03-15

### Moved AI conversation titles to runtime-backed Pi sessions

**Change:** Reworked conversation auto-title refinement so it uses a hidden ephemeral Pi runtime session, matching the memory summarization pattern, instead of invoking the Pi CLI directly for a one-off prompt.

## 2026-03-14

### Added extension registry npm auto-discovery

**Change:** Updated the standalone `extension-registry` sync flow so it can auto-discover published npm extensions instead of relying only on manually curated package names in `data/registry.json`.

**Why:** Newly published Chatons npm extensions were valid and publicly available on npm, but they never appeared in `marketplace.chatons.ai` unless someone manually added the package name to the registry source file. That created drift between npm and the marketplace catalog.

**Verified files:**
- `extension-registry/lib/discovery.ts` -- Added npm search-based discovery and candidate verification
- `extension-registry/lib/sync.ts` -- Merged curated and auto-discovered npm packages during catalog generation
- `extension-registry/lib/sync-runner.ts` -- Enabled discovery for local sync runs
- `extension-registry/lib/dev-server.ts` -- Enabled discovery for local manual syncs
- `extension-registry/api/cron/sync.ts` -- Enabled discovery for production cron syncs
- `extension-registry/lib/types.ts` -- Added `autoDiscoveredNpm` support to the registry source shape
- `extension-registry/README.md` -- Documented the new discovery behavior and curated fallback role

**Notes:**
- Builtin extensions remain explicitly declared in `data/registry.json`.
- Curated npm entries still serve as a fallback for packages that npm search has not indexed yet.
- Auto-discovered packages are merged at sync time and are not written back into `data/registry.json`.
