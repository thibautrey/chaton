# Documentation Audit

## 2026-03-20 (Latest)

### Added Kubernetes-oriented cloud service scaffold

**Change:** Added an initial cloud service scaffold for `cloud-api`, `cloud-realtime`, and `runtime-headless`, plus shared domain/protocol packages, Dockerfiles, Kubernetes base manifests, and a Kubernetes deployment reference page.

**Why:** The cloud architecture now targets Kubernetes rather than Docker Compose as the primary deployment model. The service split, probes, ingress, autoscaling, and configuration boundaries need to be encoded directly in the repository so the cloud implementation grows in a Kubernetes-native way from the start.

**Verified files:**
- `apps/cloud-api/server.ts` -- Minimal HTTP control-plane scaffold with bootstrap and health endpoints
- `apps/cloud-realtime/server.ts` -- Minimal realtime service scaffold with token and health endpoints
- `apps/runtime-headless/server.ts` -- Minimal remote runtime service scaffold with health and session-creation endpoints
- `packages/domain/index.ts` -- Shared cloud domain contracts
- `packages/protocol/index.ts` -- Shared HTTP and realtime protocol envelopes
- `packages/sdk-cloud/index.ts` -- Minimal typed SDK client for cloud APIs
- `k8s/base/*.yaml` -- Namespace, config, deployments, services, ingress, and HPAs
- `docs/content/cloud-kubernetes.mdx` -- Kubernetes deployment reference for the cloud architecture
- `docs/content/meta.json` -- Added the Kubernetes deployment page to docs navigation
- `docs/content/index.mdx` -- Added the Kubernetes deployment page to the docs landing page

## 2026-03-20

### Added proposed cloud architecture for organization-owned remote runtimes

**Change:** Added a dedicated cloud architecture design document covering multi-organization tenancy, organization-scoped provider credentials, headless server runtimes, HTTP plus WebSocket synchronization, desktop OAuth connection flow, monorepo layout, scaling model, and phased implementation guidance.

**Why:** Chatons currently documents only its desktop-first local runtime model. The planned cloud feature requires a separate target-state architecture that makes one rule explicit: cloud projects do not use local Pi, and cloud provider configuration and secrets are owned by the organization in the cloud rather than by each desktop client.

**Verified files:**
- `docs/content/cloud-architecture.mdx` -- New proposed architecture reference for cloud projects, remote runtimes, organization permissions, and realtime sync
- `docs/content/meta.json` -- Added the new cloud architecture page to documentation navigation
- `docs/content/index.mdx` -- Added the new cloud architecture page to the docs landing page

## 2026-03-20

### Hardened conversation worktrees, Git status caching, and window/runtime cleanup

**Change:** Tightened project-thread worktree lifecycle, corrected Git status cache semantics, switched worktree removal to prefer native `git worktree remove --force`, cleared stale main-window references on real window destruction, and disposed workspace runtime event subscriptions during shutdown. Pi auth/config writes were also moved to atomic JSON writes for consistency with the rest of the managed Pi directory.

**Why:** The previous worktree flow could persist an empty fallback directory as a valid conversation worktree after creation failures, direct directory deletion could leave orphaned native Git worktree metadata behind, and the Git status cache could return incorrect staged/unstaged answers depending on call order. Separately, the main process could retain a destroyed window reference while the app stayed alive in the background, and workspace handler shutdown did not release its Pi runtime subscription.

**Verified files:**
- `electron/ipc/workspace.ts` -- Worktree creation now fails closed, cleanup matches the real worktree directory layout, and async Git repo checks are awaited correctly
- `electron/ipc/workspace-handlers.ts` -- Worktree removal now receives the parent repo path and runtime event subscriptions are disposed on shutdown
- `electron/lib/git/git-service.ts` -- Reworked status caching, added native worktree removal helper, and exported parsing helpers for tests
- `electron/lib/git/git-service.test.ts` -- Added tests covering independent staged vs unstaged status parsing
- `electron/db/repos/conversations.ts` -- Allows explicit nullable runtime-field clears instead of only COALESCE-style updates
- `electron/ipc/workspace-pi.ts` -- Uses atomic JSON writes for `auth.json` sync/bootstrap paths
- `electron/main.ts` -- Registers window IPC once and clears `mainWindow` on actual window destruction
- `docs/content/pi-integration.mdx` -- Documented strict worktree persistence and native worktree removal behavior
- `docs/content/developer-guide.mdx` -- Documented stricter worktree/runtime cleanup and destroyed-window handling

## 2026-03-17 (Latest)

### Fixed CronJob removeAllListeners() TypeError in cron scheduler

**Change:** Removed calls to `CronJob.removeAllListeners()` in `electron/extensions/runtime/cron-scheduler.ts` `stop()` and `stopAll()` methods.

**Why:** The `cron` package v3.5.0's `CronJob` class does not have a `removeAllListeners()` method. This method existed in earlier versions but was removed. Calling it caused a `TypeError: task.job.removeAllListeners is not a function` during extension runtime shutdown, preventing graceful cleanup of cron scheduler automation tasks.

**Impact:** Cron-scheduled automation tasks now properly stop and cleanup without errors during app shutdown. The `stop()` method still calls `task.job.stop()` to properly halt the job; EventEmitter listener cleanup is no longer needed or attempted.

**Verified files:**
- `electron/extensions/runtime/cron-scheduler.ts` -- Removed two `task.job.removeAllListeners()` calls from `stop()` and `stopAll()` methods
- `dist-electron/extensions/runtime/cron-scheduler.js` -- Compiled output verified to have no `removeAllListeners` references

## 2026-03-17

### Routed GitHub Copilot to Anthropic message shaping

**Change:** Switched the built-in GitHub Copilot provider preset and OAuth provider bootstrap from Pi's `openai-completions` transport to `anthropic-messages`.

**Why:** Runtime verification showed that Copilot Claude models succeed with the same token, base URL, and required Copilot headers when they receive Anthropic-style payloads, while the OpenAI-compatible chat-completions path still returned `400 Bad Request` for minimal prompts in Chatons conversations. The durable fix is therefore to keep the Copilot endpoint and auth flow, but use the request format Copilot reliably accepts for these Claude-backed models.

**Verified files:**
- `electron/ipc/workspace-handlers.ts` -- Bootstraps OAuth-backed GitHub Copilot providers with `anthropic-messages`
- `src/features/workspace/provider-presets.ts` -- Defaults the GitHub Copilot preset to `anthropic-messages`
- `src/components/model/ProviderSetupForm.tsx` -- Allows `anthropic-messages` in provider setup forms
- `src/components/sidebar/settings/sections/ProvidersModelsSection.tsx` -- Allows `anthropic-messages` in provider settings state
- `src/components/onboarding/OnboardingFlow.tsx` -- Allows `anthropic-messages` in onboarding provider setup state
- `docs/content/pi-integration.mdx` -- Documented the GitHub Copilot transport change

## 2026-03-16

### Preserved nested transport details for generic provider connection failures

**Change:** Updated the Pi runtime integration docs to note that Chatons now enriches generic provider `Connection error.` failures with the nested SDK transport cause when one is available.

**Why:** Some OpenAI-compatible SDK failures collapse to a plain `Connection error.` message, which hides the actionable reason users need to diagnose local-network providers, DNS mistakes, TLS issues, and refused connections.

**Verified files:**
- `electron/pi-sdk-runtime.ts` -- Enriches generic connection failures from nested SDK causes and rewrites the last errored assistant message before snapshot emission
- `docs/content/pi-integration.mdx` -- Documented the new runtime error-surfacing behavior for provider transport failures

## 2026-03-15

### Moved AI conversation titles to runtime-backed Pi sessions

**Change:** Reworked conversation auto-title refinement so it uses a hidden ephemeral Pi runtime session, matching the memory summarization pattern, instead of invoking the Pi CLI directly for a one-off prompt.

**Why:** The CLI path could fail independently from the runtime path used by normal conversations, especially on auth-sensitive providers. Using the same runtime-backed flow keeps model selection, credential loading, and packaged-app behavior consistent with the rest of Chatons.

**Verified files:**
- `electron/ipc/workspace-title.ts` -- Replaced CLI-based title generation with hidden runtime-session title generation
- `electron/ipc/workspace.ts` -- Wired title generation to `PiSessionRuntimeManager` and hidden conversation creation
- `electron/ipc/workspace-handlers.ts` -- Passed project context through the title-generation flow
- `AGENTS.md` -- Documented hidden ephemeral Pi sessions for internal LLM tasks
- `docs/content/pi-integration.mdx` -- Documented runtime-backed hidden sessions for memory and title refinement
- `docs/content/developer-guide.mdx` -- Clarified when internal LLM tasks should avoid the CLI path

## 2026-03-14

### Added extension registry npm auto-discovery

**Change:** Updated the standalone `extension-registry` sync flow so it can auto-discover published npm extensions instead of relying only on manually curated package names in `data/registry.json`.

**Why:** Newly published Chatons npm extensions were valid and publicly available on npm, but they never appeared in `marketplace.chatons.ai` unless someone manually added the package name to the registry source file. That created drift between npm and the marketplace catalog.

**Verified files:**
- `extension-registry/lib/discovery.ts` -- Added npm search-based discovery and candidate verification
- `extension-registry/lib/sync.ts` -- Merged curated and auto-discovered npm packages during catalog generation
- `extension-registry/lib/sync-runner.ts` -- Enabled discovery for local sync runs
- `extension-registry/lib/dev-server.ts` -- Enabled discovery for local manual syncs
- `extension-registry/api/cron/sync.ts` -- Enabled discovery for production cron syncs
- `extension-registry/lib/types.ts` -- Added `autoDiscoveredNpm` support to the registry source shape
- `extension-registry/README.md` -- Documented the new discovery behavior and curated fallback role

**Notes:**
- Builtin extensions remain explicitly declared in `data/registry.json`.
- Curated npm entries still serve as a fallback for packages that npm search has not indexed yet.
- Auto-discovered packages are merged at sync time and are not written back into `data/registry.json`.
